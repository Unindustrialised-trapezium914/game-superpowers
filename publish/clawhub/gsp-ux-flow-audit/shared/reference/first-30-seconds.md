# First 30 Seconds Design Rules

A strong build earns trust quickly.
This matters for both `first-playable` and `polished-prototype`, though the bar is higher for the latter.

## What the player should understand quickly
- what they control
- what the core action is
- what counts as danger or failure
- what counts as reward or progress

## Good defaults
- get to action fast
- use one obvious primary interaction first
- keep the first objective concrete
- show only critical UI at first
- let the player learn by doing quickly

## Anti-patterns
- front-loading too many systems
- dense menus before interaction
- vague or floaty feedback
- unclear first failure state
- beautiful but unreadable HUD
